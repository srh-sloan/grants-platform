{# ── 1. What GrantOS is and the problem it solves ── #}

About GrantOS

GrantOS is a reusable government grants platform. It guides applicants through complex fund requirements, helps assessors triage consistently at volume, and generates draft monitoring plans after award — powered by AI that supports human judgement rather than replacing it.

The problem

Applying for a government grant is one of the hardest administrative tasks a small charity faces. Assessing those applications at volume is equally painful for the officials who receive them. GrantOS addresses both through one connected platform.

For applicants

A small VCFS organisation — income under £5m, 3+ years' delivery — but no dedicated bid writer. They must interpret a ten-section prospectus, check eligibility across six interlocking conditions, write to seven weighted criteria, and secure a signed endorsement letter from the right local authority officer. All within a five-week window, alongside managing a frontline service.

{{ govukInsetText({ "text": "Good projects get rejected because the application was not written well — not because the work was not fundable." }) }}

For assessors

The VCS team in MHCLG's Homelessness and Rough Sleeping Directorate must cross-reference free-text narrative against seven criteria and calculate weighted scores manually across many applications. No consistent view of evidence found versus evidence missing per criterion. Borderline applications require moderation — adding coordination overhead to an already compressed April-to-June timeline.

{{ govukInsetText({ "text": "Time spent on data wrangling is time not spent on the human judgement that actually matters in a funding decision." }) }}
{# ── 2. The five platform surfaces ── #}

Five platform surfaces

GrantOS is not a bespoke EHCF tool. It is a grant-agnostic platform: eligibility rules, criteria, weights, and form fields all live in JSON configuration, not in Python. Adding a second grant means editing a data file — not rewriting the application.

  1. Grant Builder {{ govukTag({"text": "Admin", "classes": "govuk-tag--grey"}) }}

    Department staff design a fund round — objectives, eligibility rules, criteria, weights, document requirements. No code required.

  2. Applicant Workspace {{ govukTag({"text": "Applicant", "classes": "govuk-tag--blue"}) }}

    Guided, multi-page form with eligibility pre-check, save and return, document uploads, AI-suggested draft answers, and a task-list checklist before submit.

  3. Assessor Workbench {{ govukTag({"text": "Assessor", "classes": "govuk-tag--yellow"}) }}

    Application queue with filters, side-by-side answers and scoring form, AI-assisted criterion scores with evidence snippets, override capability, and allocation dashboard.

  4. Monitoring Generator {{ govukTag({"text": "Grant team", "classes": "govuk-tag--yellow"}) }}

    Approved applications get a draft KPI pack: outputs, outcomes, baselines, reporting cadence, evidence types, milestone table. Human reviews before confirming.

  5. Portfolio Dashboard {{ govukTag({"text": "Leadership", "classes": "govuk-tag--grey"}) }}

    Director view across all active funds: application pipeline, assessor workload, missing-document count, 0-score risk flags, average turnaround.

{# ── 4. The AI layer ── #}

AI layer

Every AI feature in GrantOS surfaces information, suggests content, and highlights risk. Every output is reviewed and confirmed by a human before it becomes part of the record.

{{ govukInsetText({ "html": "AI does not auto-approve, auto-reject, or auto-submit anything. AI output is always labelled as provisional. The system records when an assessor overrides a score, preserving the AI's suggestion alongside the human's decision for audit." }) }}

1. Pre-fill — draft answers from your own documents

The applicant uploads their annual report, project plan, budget, or LA letter. The AI reads the grant's form schema and suggests answers for each field with a clear label showing which source document the suggestion came from. The applicant edits, confirms, and owns every word before submission.

2. Scoring triage — provisional criterion scores with evidence

After submission, the AI reads the application against the grant's scoring rubric — all criteria and weights from the config JSON, not hardcoded. It returns a provisional score (0–3) per criterion, confidence level, evidence found, evidence missing, and a 0-score risk flag. The assessor reviews, overrides where needed, and records a final score. If they override, both values are stored.

3. Monitoring plan — draft KPI pack for approved applications

For a funded application, the AI generates a draft monitoring plan: KPI name, definition, evidence type, reporting cadence, owner, and milestone table — all shaped by the approved application answers and the grant's monitoring template. A grant manager reviews and confirms before the plan is issued to the grantee.

{# ── 5. Hackathon challenges ── #}

How GrantOS addresses all four hackathon challenges

Challenge 1: From PDF to digital service

The EHCF prospectus is a ten-section GOV.UK guidance page. GrantOS turns it into a structured, guided application journey with eligibility pre-check, conditional logic, task-list checklist, and submission confirmation — all driven by JSON configuration that a policy team can update without touching code.

  • Grant Builder UI
  • Task-list form runner
  • Eligibility pre-check from config
  • Save and return

Challenge 2: Unlocking the dark data

An applicant's annual report, project plan, and budget contain all the evidence MHCLG needs — but it sits unstructured in PDFs and spreadsheets. The AI pre-fill feature reads those documents, extracts relevant evidence, and maps it to form fields — making the dark data accessible to both applicant and assessor, with provenance tracked.

  • Document ingestion
  • AI pre-fill with source attribution
  • Knowledge packs
  • Grant schema extractor (stretch)

Challenge 3: Supporting casework decisions

Assessors face volume pressure and inconsistent free-text. The Assessor Workbench gives them a structured view of every application: eligibility gate, per-criterion evidence summary, provisional scores, 0-score risk flags, and missing-evidence detection — all as a starting point for their own informed judgement, not a replacement for it.

  • Eligibility gate
  • AI-assisted scoring with evidence snippets
  • 0-score risk flags
  • Override and audit trail

Challenge 4: Knowing your own organisation

Senior leaders in MHCLG's Homelessness team need portfolio visibility — across funds, assessors, and timelines — that currently lives in disconnected spreadsheets. The Portfolio Dashboard gives them application pipeline, assessor workload, common rejection patterns, KPI submission status, and cross-fund comparability in a single view.

  • Portfolio dashboard
  • Director view
  • KPI monitoring
  • Cross-fund data
{# ── 6. The scoring model ── #}

Scoring model

The EHCF scoring model is not baked into Python. It lives in seed/grants/ehcf.json. Changing a weight, adding a criterion, or adjusting the auto-reject rule means editing that file — the scoring engine reads from config on every calculation.

Each criterion is scored 0–3 (0 = unacceptable, 1 = satisfactory, 2 = good, 3 = excellent). The formula is:

weighted_total = sum of (score × weight) for each criterion — maximum 300

{# ── 8. Demo journey ── #}

Demo journey

Every demo user is seeded in the database. No manual sign-up required. The platform is ready to demo immediately.

  1. Grant Admin {{ govukTag({"text": "Admin", "classes": "govuk-tag--grey"}) }}

  3. Assessor {{ govukTag({"text": "Assessor", "classes": "govuk-tag--yellow"}) }}
