# Session Note — 2026-04-16

## What was done

Updated the orchestrate skill to support parallel orchestrator sessions running
in separate chats without colliding.

### Changes made

1. **`.claude/skills/orchestrate/SKILL.md`** — rewrote with two key changes:
   - New **Step 0** (claims file protocol): orchestrator reads
     `.claude/orchestrator-claims.json` on startup to see tasks claimed by
     other sessions; writes its own claims immediately after plan approval and
     before launching any sub-agents; clears claims after PRs are created.
   - **Sub-agent limit raised from 3 → 5**: selection rule now allows up to 5
     parallel agents (all four stream slots + a docs/config fifth slot).

2. **`.claude/orchestrator-claims.json`** — seeded as `{"claimed":[]}` so the
   first orchestrator run finds a clean file.

3. **`CLAUDE.md`** — orchestrator workflow section updated to describe the
   claims mechanism and the 5-agent limit.

4. **`CHANGELOG.md`** — entry added under 2026-04-16 for both changes.

## Decisions

- Claims are keyed by the exact `task` string from PLAN.md to make matching
  unambiguous across sessions.
- Stale claim cleanup threshold set to 2 hours to handle crashed sessions.
- Claims are removed after PRs are open (not after merge) — PLAN.md `[~]`
  markers serve as the durable done signal.

## Follow-ups

- The claims protocol assumes orchestrator sessions are roughly time-aligned.
  If a session crashes mid-dispatch, claims will persist until the 2-hour
  stale-cleanup threshold. Could add a `--force-clear-claims` escape hatch
  if this becomes a problem.
- Five agents is the new max, but in practice it depends on how many truly
  independent tasks remain. The selection rules still enforce one-per-stream.
